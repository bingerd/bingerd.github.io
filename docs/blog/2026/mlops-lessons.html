MLOps Lessons Learned Shipping Real Models

After years of shipping ML models to production, certain patterns keep emerging. These aren't textbook lessons — they're hard-won insights from late-night pages, silent model failures, and pipelines that looked fine until they weren't.

1. CI/CD for ML Is Not Software CI/CD

Traditional CI/CD tests code correctness. ML CI/CD must also verify:

  • Data quality: Schema validation, distribution checks, and freshness assertions before training starts.
  • Model quality gates: Automated evaluation against a held-out set with minimum metric thresholds.
  • Reproducibility: Pinned dependencies, deterministic seeds, and versioned training data.

We learned this the hard way when a model passed all unit tests but shipped with a training set that had drifted 40% from the validation distribution.

# Example: quality gate in CI pipeline
def model_quality_gate(model, eval_data, min_auc=0.85):
    auc = evaluate(model, eval_data)
    assert auc >= min_auc, f"AUC {auc:.3f} below threshold {min_auc}"
    log_metric("model_auc", auc)

2. Model Registries Are Non-Negotiable

Every model artifact needs a single source of truth. A model registry should track:

  • Model version, training date, and data snapshot used
  • Evaluation metrics at training time
  • Deployment status (staging, canary, production, retired)
  • Lineage: which pipeline, which commit, which experiment
"If you can't answer 'what model is running in production right now and why?' in under 30 seconds, your MLOps isn't ready."

3. Monitoring Beyond Accuracy

Accuracy metrics only tell you the model worked on yesterday's data. In production, you need:

  • Input drift detection: Statistical tests (KS, PSI) on feature distributions at inference time.
  • Prediction distribution monitoring: If your model suddenly predicts 95% one class, something broke.
  • Latency percentiles: p50 is misleading — track p95 and p99 for real user experience.
  • Data pipeline freshness: A model serving stale features is often worse than no model at all.

4. Incident Response for ML Systems

ML incidents are different from software incidents. The failure mode is often silent degradation rather than a crash:

  • Runbook for model rollback: Have a one-command rollback to the last known-good model version.
  • Feature kill switches: Disable individual features without redeploying the model.
  • Fallback strategies: When the model is down, serve a rules-based heuristic or cached predictions.
  • Blameless post-mortems: ML failures often span data, training, and serving — assign owners to each layer.

5. Team Workflows That Scale

As the team grows, ML workflows need structure:

  • Experiment tracking: Every training run logged with hyperparameters, metrics, and artifacts. No exceptions.
  • Feature ownership: Each feature in the feature store has a clear owner and SLA.
  • Model review process: Just like code review — a second pair of eyes on evaluation results, data splits, and deployment config before any production push.

Conclusion

MLOps maturity isn't about tooling — it's about operational discipline. The teams that ship reliably are the ones that treat models like services: versioned, monitored, tested, and always one command away from a safe rollback.
